## Setup & how to run

### Prerequisites

- PHP **8.3+**, Composer
- **MySQL 8+** — create an empty database before running migrations:

  ```sql
  CREATE DATABASE order_payment_api;
  ```

### Steps

```bash
# 1. Install PHP dependencies
composer install

# 2. Create your env file
cp .env.example .env

# 3. Set your MySQL credentials in .env (see below), then:

# 4. Generate the app key
php artisan key:generate

# 5. Generate the JWT secret (writes JWT_SECRET to .env)
php artisan jwt:secret

# 6. Run migrations
php artisan migrate

# 7. Start the dev server
php artisan serve
```

The API is now available at:

```
http://localhost:8000/api
```

> **Base URL note:** all routes live under the `/api` prefix. The Postman collection uses a
> `baseUrl` variable that should be set to `http://localhost:8000/api/` (with the trailing slash).

## Environment configuration

Relevant `.env` keys:

```dotenv
# Database (MySQL)
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=order_payment_api
DB_USERNAME=root
DB_PASSWORD=your_password

# JWT (generated by `php artisan jwt:secret`)
JWT_SECRET=...

# Stripe gateway credentials (read by the Stripe payment strategy)
STRIPE_KEY=
STRIPE_SECRET=
```

Set `DB_DATABASE`, `DB_USERNAME`, and `DB_PASSWORD` to match the MySQL database you created.

Gateway credentials are resolved through `config/services.php` → `services.stripe.*`, so they are
configured purely via `.env` and never hardcoded.

## Authentication

The API uses **JWT bearer tokens**. Register or log in to obtain a token, then send it on every
protected request:

```
Authorization: Bearer <token>
```

1. `POST /api/auth/register` or `POST /api/auth/login` → returns a `token`.
2. Send `Authorization: Bearer <token>` (the Postman collection stores it in a `token` variable).

## API reference

Base URL: `http://localhost:8000/api`

All responses follow the envelope:

```json
{ "success": true, "message": "…", "data": { } }
```

### Authentication

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| POST | `/auth/register` | — | Register and receive a token |
| POST | `/auth/login` | — | Log in and receive a token |
| GET | `/auth/me` | ✅ | Current authenticated user |
| DELETE | `/auth/logout` | ✅ | Invalidate the current token |
| POST | `/auth/refresh` | ✅ | Issue a fresh token |

**Register / login body**

```json
{
  "name": "Mohammed Hammad",
  "email": "mohammed@test.com",
  "password": "Qwertyuiop@1",
  "password_confirmation": "Qwertyuiop@1"
}
```

### Orders

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| GET | `/orders` | ✅ | List the user's orders (paginated). Filter with `?status=pending\|confirmed\|cancelled` |
| POST | `/orders` | ✅ | Create an order from line items |
| GET | `/orders/{order}` | ✅ | Show one order (with items + payment) |
| PUT/PATCH | `/orders/{order}` | ✅ | Update status and/or replace items |
| DELETE | `/orders/{order}` | ✅ | Delete (only if no payment) |

**Create order** — `POST /orders`

```json
{
  "items": [
    { "product_name": "test product", "quantity": 2, "price": 40 }
  ]
}
```

Response `201`:

```json
{
  "success": true,
  "message": "Order created successfully.",
  "data": {
    "id": 1,
    "user_id": 1,
    "status": "pending",
    "total": "80.00",
    "items": [
      { "id": 1, "order_id": 1, "product_name": "test product", "quantity": 2, "price": "40.00", "subtotal": "80.00" }
    ]
  }
}
```

**Update order** — `PATCH /orders/{order}` (both fields optional)

```json
{ "status": "confirmed" }
```

```json
{ "items": [ { "product_name": "Monitor", "quantity": 1, "price": 250 } ] }
```

### Payments

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| GET | `/payments` | ✅ | List all the user's payments (paginated) |
| GET | `/orders/{order}/payment` | ✅ | Show the payment for an order (404 if none) |
| POST | `/orders/{order}/payment` | ✅ | Process a payment for a **confirmed** order |

**Process payment** — `POST /orders/{order}/payment`

```json
{ "payment_method": "stripe" }
```

Response `201`:

```json
{
  "success": true,
  "message": "Payment processed successfully.",
  "data": {
    "id": 1,
    "order_id": 1,
    "method": "stripe",
    "status": "successful",
    "amount": "80.00",
    "transaction_reference": "pi_xxxxxxxxxxxxxxxxxxxxxxxx",
    "paid_at": "2026-06-25T12:00:00.000000Z"
  }
}
```

### Common error responses

| Status | When |
|--------|------|
| `401 Unauthorized` | Missing/invalid JWT |
| `403 Forbidden` | Acting on another user's order |
| `404 Not Found` | Order/payment does not exist |
| `409 Conflict` | Deleting an order that has a payment |
| `422 Unprocessable Entity` | Validation errors, paying a non-confirmed order, paying twice |

Validation errors use Laravel's standard shape:

```json
{ "message": "The items field is required.", "errors": { "items": ["The items field is required."] } }
```

## Adding a new payment gateway (extensibility)

Gateways use the **Strategy pattern** behind a **Factory**. Adding one (e.g. PayPal) is three small
steps — no controller, service, or business-logic changes required.

**1. Add the method to the enum** — `app/Enums/PaymentMethod.php`

```php
enum PaymentMethod: string
{
    case CashOnDelivery = 'cash_on_delivery';
    case Stripe         = 'stripe';
    case PayPal         = 'paypal'; // 👈 new
}
```

**2. Create the strategy** — `app/Payments/PayPalStrategy.php`

```php
namespace App\Payments;

use App\Enums\PaymentStatus;
use App\Models\Payment;

class PayPalStrategy implements PaymentStrategy
{
    public function pay(Payment $payment): void
    {
        // Credentials come from .env via config('services.paypal.*')
        $payment->update([
            'status'                => PaymentStatus::Successful,
            'transaction_reference' => 'pp_'.uniqid(),
            'paid_at'               => now(),
        ]);
    }
}
```

**3. Register it in the factory** — `app/Payments/PaymentStrategyFactory.php`

```php
return match ($method) {
    PaymentMethod::CashOnDelivery => app(CashOnDeliveryStrategy::class),
    PaymentMethod::Stripe         => app(StripeStrategy::class),
    PaymentMethod::PayPal         => app(PayPalStrategy::class), // 👈 new
};
```

(Optional) add credentials to `config/services.php` + `.env`:

```php
'paypal' => [
    'client_id' => env('PAYPAL_CLIENT_ID'),
    'secret'    => env('PAYPAL_SECRET'),
],
```

Each strategy is resolved from the service container, so it can declare its own dependencies
(HTTP clients, SDKs) in its constructor. The `PaymentService` and `PaymentController` stay untouched.

## Postman collection

A ready-to-import collection is included at the project root:

**[`api.postman_collection.json`](./api.postman_collection.json)**

It is organized by functionality — **auth**, **orders**, **payments** — and uses two variables:

| Variable | Example value |
|----------|---------------|
| `baseUrl` | `http://localhost:8000/api/` |
| `token` | the JWT returned by register/login |

**Usage**

1. Import `api.postman_collection.json` into Postman.
2. Set the `baseUrl` collection variable to `http://localhost:8000/api/`.
3. Run **auth → 01 create account** (or **02 User login**) and copy the returned token into the
   `token` variable. The collection sends it as a Bearer token automatically.
4. Exercise the **orders** and **payments** requests (remember: confirm an order before paying it).

## Architecture & notes

The codebase is layered for clarity and DRY-ness:

- **Controllers** (`app/Http/Controllers/API`) — thin; handle HTTP and authorization only.
- **Form Requests** (`app/Http/Requests`) — validation + meaningful messages.
- **Actions** (`app/Actions`) — orchestrate a single use case in one transaction.
- **Services** (`app/Service`) — data/persistence and business operations.
- **Payments** (`app/Payments`) — `PaymentStrategy` interface, gateway strategies, and the factory.
- **Resources** (`app/Http/Resources`) — JSON shaping.
- **Policies** (`app/Policies`) — per-order ownership authorization.

Order status lifecycle: `pending → confirmed → cancelled`. A typical flow is: create order
(`pending`) → update to `confirmed` → process payment.

## Assumptions

- **Order items are free-form** (`product_name`, `quantity`, `price`) per the task; there is no
  separate product catalog or stock tracking.
- **"All payments"** (`GET /payments`) is scoped to the authenticated user's own orders rather than
  every user's, for security.
- The **Stripe gateway is simulated** — it reads `STRIPE_SECRET` from config and returns a generated
  `pi_…` reference instead of calling the real Stripe API. The strategy is the single place to drop
  in a real SDK call.
- **Cash on delivery** stays `pending` (collected on delivery); **Stripe** resolves to `successful`.
- Orders use **soft deletes**, so a deleted order is retained (with `deleted_at`) and excluded from
  listings/lookups.
